# 当前实现与目标架构差异

日期：2026-03-19

## 当前已经落地

- 主执行链路是 `core/workflows/voucher_pipeline.py` 的函数式 pipeline
- 规则修正、借贷校验、人工复核、JSON 导出已经形成闭环
- 知识层已能解析制度 PDF 与科目表 XLS，并提供结构化搜索回退
- 前端已把中间过程可视化到任务详情页

## 目标架构中尚未完全落地的部分

### 1. LangGraph

- 当前只有流程分层与流程图，尚未重构为真正的 `StateGraph`
- 这不影响当前交付链路，但影响后续 agent 化扩展能力

### 2. LanceDB + Embedding

- 当前优先依赖 JSON / FTS 回退搜索
- `bge-m3` 模型路径已确定，但 embedding 召回尚未正式进入主链路

### 3. 轻量知识图谱

- 当前阶段不是必须项
- 后续如需增强“事项 -> 制度条款 -> 科目路径”的解释能力，可在规则层稳定后加入

## 现阶段判断

- 当前实现适合先完成 `方案1`：以“可视化 + 强校验 + 人工阻断”确保结果可控
- 不建议现在就大规模重构到方案3，否则会拖慢当前可交付版收尾
- 更合理的路线是：
  1. 先做完 `方案1` 收尾
  2. 再把规则库、embedding 检索、LangGraph 编排逐层替换进去

## 下一步重构顺序

1. 继续沉淀 `split_rules` 为通用规则注册表
2. 补多票据类型回归集
3. 接入真实 embedding 召回
4. 将 pipeline 改造成状态机式工作流
5. 最后再评估是否引入轻量知识图谱
